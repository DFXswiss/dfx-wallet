# WDK Compliance Audit

This document captures the gap between our current WDK integration and the
canonical [Wallet Development Kit documentation](https://docs.wdk.tether.io)
as of **May 2026**.

It is intentionally a snapshot, not a living spec. Re-run the audit when we
plan a WDK upgrade.

## TL;DR

We use the right WDK packages, but every one of them is behind npm `latest`,
and the integration shape (`WdkAppProvider` props, bundle generation, hook
ergonomics) reflects an older snapshot of the SDK. The biggest single gap is
the **bundle architecture**: the docs treat `@tetherto/pear-wrk-wdk` as a
prototype-only fallback and assume a custom `.wdk/` bundle generated by
`@tetherto/wdk-worklet-bundler` for production.

## Findings

### 1. Bundle architecture ‚Äî non-compliant

The docs ([Worklet Bundler](https://docs.wdk.tether.io/tools/worklet-bundler))
specify the production path as:

```bash
npm install -g @tetherto/wdk-worklet-bundler
wdk-worklet-bundler init && wdk-worklet-bundler generate
```

producing a `.wdk/` directory imported as `import { bundle } from './.wdk'`.

We have **none** of: `@tetherto/wdk-worklet-bundler` installed, a `.wdk/`
directory, or a `wdk.config.js`. We rely on the github-pinned
`@tetherto/pear-wrk-wdk` instead ‚Äî which the
[React Native Core docs](https://docs.wdk.tether.io/tools/react-native-core)
explicitly call out as the prototype path: *"includes all blockchain modules,
resulting in a larger bundle size"*.

### 2. `WdkAppProvider` API ‚Äî non-compliant

Docs:

```tsx
<WdkAppProvider bundle={{ bundle }} wdkConfigs={configs}>
```

Our `app/_layout.tsx`:

```tsx
<WdkAppProvider networkConfigs={getNetworkConfigs()} tokenConfigs={getTokenConfigs()}>
```

The pinned `wdk-react-native-core` (commit `0b2be6eb`) physically does not
accept `bundle` or `wdkConfigs` props; its `WdkAppProviderProps` only takes
`networkConfigs` / `tokenConfigs` / `currentUserId` / `enableAutoInitialization`.

Migration path requires bumping `wdk-react-native-core` first, not just
renaming props.

### 3. `wdk.config.js` shape ‚Äî non-compliant

Docs ([Configuration](https://docs.wdk.tether.io/tools/worklet-bundler/configuration)):

```js
networks: {
  ethereum: { package: '@tetherto/wdk-wallet-evm-erc-4337' },
  bitcoin:  { package: '@tetherto/wdk-wallet-btc' }
}
```

We have nothing equivalent. `src/config/chains.ts` only carries the runtime
config (`chainId`, `provider`, `bundlerUrl`, ‚Ä¶), which would still be passed
as `wdkConfigs` to the provider, but the *bundle config* is missing entirely.

### 4. Hooks ‚Äî partially compliant

Per the [Core docs](https://docs.wdk.tether.io/tools/react-native-core), the
public hooks are `useWdkApp`, `useWalletManager`, `useAccount`, `useAddresses`,
`useBalance`, `useBalancesForWallet`, `useRefreshBalance`.

| Hook | Used? | Notes |
|---|---|---|
| `useWdkApp()` | ‚úÖ used in `verify.tsx` | Returns old `{ isReady, isInitializing }` shape, not the new `{ state: { status: 'INITIALIZING' \| 'NO_WALLET' \| 'LOCKED' \| 'READY' \| 'ERROR' } }` machine |
| `useWalletManager()` | ‚úÖ used in 9 files | We call `initializeFromMnemonic`, `deleteWallet`, `wallets`, `activeWalletId`, `initializeWallet`. Docs say the API is `createWallet(id)`, `unlock()`, `lock()`, `deleteWallet()` ‚Äî verbs and return shape differ |
| `useAccount({ network, accountIndex })` | ‚ùå never used | This is the documented "actor" hook for `address`, `send`, `sign`, `verify`, `estimateFee`. We route signing through the lower-level worklet directly via `useSendFlow.ts` |
| `useAddresses()` | ‚ùå never used | We pull addresses out of `useWallet().wallets[id].addresses` instead |
| `useBalance()` | ‚ùå never used | ‚Äî |
| `useBalancesForWallet()` | ‚úÖ used in `dashboard.tsx` | Compatible with current docs |
| `useRefreshBalance()` | ‚úÖ used in `useSendFlow.ts` | Compatible with current docs |

### 5. `@tetherto/wdk-secret-manager` ‚Äî correctly removed

We removed `@tetherto/wdk-secret-manager` during the WDK migration. The docs
still document it as a standalone tool, but it is no longer required by
`wdk-react-native-core` because `@tetherto/wdk-react-native-secure-storage`
(pulled transitively) handles secret management inside the worklet. Removal
was correct.

### 6. Pricing packages ‚Äî OK

Both `@tetherto/wdk-pricing-provider@1.0.0-beta.3` and
`@tetherto/wdk-pricing-bitfinex-http@1.0.0-beta.2` match npm `latest`.
Usage in `src/services/pricing-service.ts` is consistent with the
[Price Rates docs](https://docs.wdk.tether.io/tools/price-rates/configuration).

### 7. Android `minSdkVersion` ‚Äî OK

`34` in `app.json`, well above the docs' required `29`.

### 8. Indexer API env vars ‚Äî wired

The [React Native Quickstart](https://docs.wdk.tether.io/start-building/react-native-quickstart)
documents:

```
EXPO_PUBLIC_WDK_INDEXER_BASE_URL=https://wdk-api.tether.io
EXPO_PUBLIC_WDK_INDEXER_API_KEY=<key>
```

These names now flow through `src/config/env.ts` and are consumed by a
typed REST client at `src/services/wdk-indexer/` (see the section below).
Balance reads still go via RPC by default; the indexer is opt-in.

#### Note: indexer is not a provider concern

The new `@tetherto/wdk-react-native-core@^1.0.0-beta.9` API surface
(`WdkAppProvider`, `useAccount`, `useBalancesForWallet`, ‚Ä¶) does not
accept an indexer configuration ‚Äî that was a feature of the **legacy**
`@tetherto/wdk-react-native-provider`'s `WalletProvider.config.indexer`
slot, which is what the public `wdk-starter-react-native` repo still
uses. Greping `node_modules/@tetherto` after the migration shows no
package consuming `WDK_INDEXER` env vars; the published
`@tetherto/wdk-indexer-http` SDK is GitHub-only.

The pragmatic shape is therefore:

- `src/config/env.ts` ‚Äî exposes `wdkIndexerBaseUrl` / `wdkIndexerApiKey`
  with a default for the URL and an empty default for the key
- `src/services/wdk-indexer/config.ts` ‚Äî `getIndexerConfig()` returns
  `{ baseUrl, apiKey } | null` (null when the API key is empty), so
  callers must treat the indexer as optional infra, not required
- `src/services/wdk-indexer/client.ts` ‚Äî `WdkIndexerClient` covers the
  two endpoints we'd need first (`/api/v1/{chain}/{token}/{address}/{token-balances|token-transfers}`)
- `src/services/wdk-indexer/network-mapping.ts` ‚Äî `ChainId` ‚Üî indexer
  blockchain slug

Nothing in the app currently calls the client. When/if we add an
on-chain transfer history screen (today's transaction history reads
from the DFX API, not the chain), the consumer plugs into this module
instead of writing it from scratch.

### 9. `bare-pack@1` workaround

Our `scripts/ensure-bare-pack.js` `preinstall` hook globally pins
`bare-pack@1` so the `@tetherto/pear-wrk-wdk` postinstall (`npx bare-pack
--target ‚Ä¶`) doesn't pick up the breaking `bare-pack@2` release. Develop
solved the same problem differently by inlining `npx bare-pack@1` into
`postinstall`, but their script targets the older
`@tetherto/wdk-react-native-provider` package which we no longer use.

The workaround disappears once we adopt `@tetherto/wdk-worklet-bundler`
(which uses `bare-pack` internally with a known-good version).

## Recommended migration plan

In commit-sized chunks, ordered for minimum risk:

1. **Bump `@tetherto/wdk-react-native-core` to `^1.0.0-beta.9`** off npm
   (drop the github commit pin). This forces the `bundle` /
   `wdkConfigs` / `useAccount` migration but cleans up the API drift.
2. **Add `@tetherto/wdk-worklet-bundler`** as a dev dependency, write
   `wdk.config.js` (only the modules we ship: `wdk-wallet-evm-erc-4337`,
   `wdk-wallet-btc`, `wdk-wallet-spark`), generate `.wdk/`, swap the
   provider over.
3. Replace `useSendFlow`'s manual signing with `useAccount().send(...)`
   and adopt `BaseAsset` for token definitions.
4. Migrate `useWalletManager` callers to `createWallet` / `unlock` /
   `lock` and the new `state.status` machine in `useWdkApp`.
5. Remove the `bare-pack@1` preinstall workaround once the bundler
   replaces `pear-wrk-wdk`'s postinstall.
6. Add the indexer env vars (or document that we intentionally skip
   indexer features for now). **Done** ‚Äî wired the env-var contract
   and a typed REST client; left unconsumed because no UI needs it
   yet.

Each step is independently shippable and worth its own ticket.
